# Workmate Test Task

Данный репозиторий был создан для выполнения тестового задания в комапнию Workmate на должность Go разработчика.

## Запуск сервиса

1. Склонировать репозиторий с помощью команды **git clone https://github.com/AeZG/workmate-test**.
2. Перейти в корень проекта и написать команду make all.
3. Подождать пока все необходимые зависимости не загрузятся и запустится сервер.
4. Перейти по ссылке **http://localhost:8080/api/v1/docs/index.html**.
5. На экране будет виден интерфейс для взаимодействия с API сервиса. Все эндпойнты подробно в нем описаны.
6. Для создания новой таски необходимо нажать на дропдаун меню **POST /task** и нажать на кнопку **Try it out**.
7. Появится кнопка **Execute**, при нажатии содастся новая таска и выведится в ответе UUID таски.
8. В GET и DELETE запросах можно удалять по UUID созданную таску, либо получать информацию о ней.
9. Для прекращения работы сервиса необходимо нажать CTRL+C в терминале, где была запущена программа.

## Объяснение кода и архитектуры решения

Данный сервис был создан по принципу трехслойной архитектруы создания сервисов.

- handler отвечает за **REST** эндпойнты и создание раутера
- service содержит в себе бизнес логику продукта
- repository слой отвечает за запросы к базам данных.

В данном случае модуль task является *"репозиторным"* слоем, так как есть мапа где хранится информация о тасках и методы для взаимодействия с ней. handler не может напрямую обрщаться к репозиторному слою и наоборот.

Данный подход был выбран для разделения зон ответсвенности функционала между частями сервиса для более легкого наращивания функционала продукта.

Unit тестирование также облегчается, так как можно создавать mocks для отдельных слоев.

Была создана *Swagger* документация для более интуитивного взаимодействия с API сервиса.

Для создания эндпойнтов был использован *go-kit* и *gorilla/mux*

Все таски работают конкурентно для корректного отсчета времени. Везде прописаны mutex locks/unlocks для предотваращения *race conditions*

По итогам, можно сказать, что задание выполнено успешно и были учтены все необходимые факторы для дальнейшего наращивания функционала. Также присутствует здоровый баланс между использованием инженерных практик и сложностью кода.
